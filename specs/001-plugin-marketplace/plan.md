## Summary

Build a private Claude Code plugin marketplace containing three independent, installable plugins that enforce infrastructure standards and enable DevOps diagnostics:

1. **terraform-standards**: Enforces tagging (Environment, Owner), kebab-case naming, and encryption on storage resources via pre-apply checklist command and commit-blocking hook for hardcoded credentials.

2. **k8s-troubleshooter**: Provides specialized agent for diagnosing Kubernetes pod failures (CrashLoopBackOff, Pending, OOMKilled) and a skill for enforcing manifest best practices (resource requests/limits, health probes).

3. **aws-security-review**: Provides skill to flag overly permissive IAM policies, public S3 buckets, and unrestricted security group access.

Each plugin is independently installable and can be adopted without requiring others. All plugins follow Claude Code plugin architecture and must provide English documentation.

**Structure Decision**: Each plugin is a self-contained directory under `plugins/<name>/` with its own `.claude-plugin/plugin.json` manifest, documentation, and implementation files. This enables:
- Independent installation (copy `plugins/<name>/` to user's Claude Code config)
- Isolated versioning (each plugin semantic version independently)
- Clear dependency boundaries (no shared code)
- Marketplace-level registry via `marketplace.json` for discovery

## Design Decisions & Rationale

### Marketplace Registry
**Decision**: Central `marketplace.json` at repository root lists all available plugins with metadata.

**Rationale**: Users can discover, understand, and install plugins without cloning the entire marketplace. Registry structure enables future automation (installation scripts, version checks).

### No State Management
**Decision**: All three plugins are stateless analysis tools (no persistent configuration or caches).

**Rationale**: Simplifies installation and reduces user burden. Analysis rules are baked into plugin logic, not external configuration files. Aligns with "independence" principle (no shared state between plugins).

### Hook-Based Enforcement (terraform-standards)
**Decision**: Pre-commit hook blocks commits with hardcoded credentials; pre-apply checklist command validates tagging and encryption.

**Rationale**: Hooks prevent accidental leaks at commit time (security boundary). Commands provide linting-style feedback before `terraform apply`. Both independent of other plugins.

### Agent + Skill Pattern (k8s-troubleshooter)
**Decision**: Agent handles interactive diagnosis; skill validates manifests in batch.

**Rationale**: Agents excel at multi-step reasoning (pod state в†’ likely causes в†’ remediation). Skills enable pre-deployment validation in CI/CD. Two entry points serve different workflows.

### Skill-Only for aws-security-review
**Decision**: Single skill that reviews provided AWS configurations (IAM, S3, security groups).

**Rationale**: Configuration review is stateless analysis. Skill is sufficient; no agent needed. Enables easy integration into audit pipelines.
